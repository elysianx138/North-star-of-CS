## 一、数组与指针的本质区别

### 数组名不是指针，但大部分时候像指针

```cpp
int arr[4] = {10, 20, 30, 40};
```

| 表达式 | 值 | 类型 | 说明 |
|:-------|:---|:----|:-----|
| `arr` | `0x...10` | `int*` | 退化为首元素地址 |
| `&arr[0]` | `0x...10` | `int*` | 首元素地址 |
| `&arr` | `0x...10` | `int(*)[4]` | 整个数组的地址 |

**三个地址值相同，但类型不同 → 指针运算步长不同！**

### 数组名 decay（退化）

在**大多数**表达式中，数组名自动退化成首元素的指针：

```cpp
int* p = arr;       // ✅ arr 退化为 int*
cout << arr;        // ✅ 退化为指针传给 cout
```

但在 **`sizeof`** 和 **`&`** 时不退化：

```cpp
sizeof(arr);        // 16（整个数组的大小，不是指针大小！）
sizeof(&arr[0]);    // 8（指针大小）
```

### 指针步长 = 指向类型的大小

```cpp
arr + 1    // 跳 4 字节（一个 int）
&arr + 1   // 跳 16 字节（整个 int[4] 数组）
```

**本质：** 指针的类型决定了一步跨多远。`arr` 是 "指向 int" → 步长 4，`&arr` 是 "指向 int[4] 数组" → 步长 16。

关键区分：

| | 数组名 `arr` | 指针 `int* p` |
|:--|:-----------|:-------------|
| 存什么 | 数组本身（一整块内存） | 一个地址 |
| 能不能改指向 | ❌ `arr++` 编译错误 | ✅ `p++` 可以 |
| `sizeof` | 整个数组大小（16） | 指针大小（8） |

### 实验 1：数组 vs 指针对比（运行 `./array_vs_ptr_test`）

**完整代码：**

```cpp
#include <iostream>
using namespace std;

int main() {
    int arr[4] = {10, 20, 30, 40};
    int* p = arr;  // decay 发生

    cout << "========== 第一组：地址值 ==========" << endl;
    cout << "arr         = " << arr         << endl;
    cout << "&arr[0]     = " << &arr[0]     << endl;
    cout << "&arr        = " << &arr        << endl;
    cout << endl;

    cout << "========== 第二组：sizeof 揭露真相 ==========" << endl;
    cout << "sizeof(arr) = " << sizeof(arr) << endl;  // 16
    cout << "sizeof(p)   = " << sizeof(p)   << endl;  // 4 或 8
    cout << endl;

    cout << "========== 第三组：步长实验 ==========" << endl;
    cout << "arr + 1     = " << arr + 1     << endl;  // +4 字节
    cout << "&arr + 1    = " << &arr + 1    << endl;  // +16 字节
    cout << "p + 1       = " << p + 1       << endl;  // +4 字节
    cout << endl;

    // 算跳了多少字节
    long long step_arr = (long long)(arr + 1) - (long long)arr;
    long long step_and = (long long)(&arr + 1) - (long long)(&arr);
    long long step_p   = (long long)(p + 1) - (long long)p;

    cout << "arr+1 - arr     = " << step_arr << " 字节" << endl;
    cout << "&arr+1 - &arr   = " << step_and << " 字节" << endl;
    cout << "p+1 - p        = " << step_p   << " 字节" << endl;
    cout << endl;

    cout << "========== 第四组：能否 ++？ ==========" << endl;
    // arr++;  // ❌ 编译错误！数组名不能改指向
    p++;  // ✅ 指针可以改指向
    cout << "p++ 之后 *p = " << *p << " (arr[1])" << endl;
    p = arr;  // 恢复

    return 0;
}
```

**编译运行：**

```bash
g++ array_vs_ptr_test.cpp -o array_vs_ptr_test
./array_vs_ptr_test
```

**输出结果：**

```
========== 第一组：地址值 ==========
arr         = 0x61fee0
&arr[0]     = 0x61fee0
&arr        = 0x61fee0

========== 第二组：sizeof 揭露真相 ==========
sizeof(arr) = 16
sizeof(p)   = 4

========== 第三组：步长实验 ==========
arr + 1     = 0x61fee4
&arr + 1    = 0x61fef0
p + 1       = 0x61fee4

arr+1 - arr     = 4 字节     ← int*，指向 int，步长 4
&arr+1 - &arr   = 16 字节    ← int(*)[4]，指向整个数组，步长 16
p+1 - p        = 4 字节     ← int*，跟 arr+1 一样

========== 第四组：能否 ++？ ==========
p++ 之后 *p = 20 (arr[1])
// 如果取消 arr++; 的注释 → 编译错误：数组名不是左值
```

**结果解读：**

1. **三个地址值相同** → `arr`、`&arr[0]`、`&arr` 都指向同一块内存的起点
2. **`sizeof(arr)` = 16 而 `sizeof(p)` = 4/8** → 这是数组不是指针的铁证
3. **步长不同** → `arr+1` 跳 4 字节，`&arr+1` 跳 16 字节——地址值相同但类型不同，跨步不同
4. **`arr++` 报错** → 数组名是内存块的名字，不是存地址的变量，不能改指向

### 底层原理：为什么要有 decay？

核心原因只有一个——**C 的设计哲学：不要隐式拷贝大数据。**

> **数组是"一大块内存"的名字，不是存地址的变量。** 如果数组名在表达式中保持原样，那么 `int* p = arr` 这样的赋值就变成了"把一整块 16 字节拷贝给指针"——不合理。

所以 C 的发明者 Dennis Ritchie 做了一个妥协：

```
int arr[4];    ← arr 本身是"一整块内存"
int* p = arr;  ← 但大多数时候你想用的是它的地址，不是它的拷贝
                → 编译器自动帮你取地址：p = &arr[0]
                → 这就是 decay
```

**为什么 sizeof 和 & 不 decay？**
- `sizeof(arr)` 如果 decay 了，就等于 `sizeof(int*)` 永远返回 4 或 8——那你就再也拿不到数组的真正大小了
- `&arr` 如果 decay 了，`&arr` = `&(&arr[0])`，变成了取指针的地址——乱套了

所以 C 的设计者决定：**这两个场景保留数组的"真身"，其余场合一律 decay**。

**一句话总结：** 数组名不是指针，但编译器在大多数场合帮你自动 decay 成指针——只有 `sizeof` 和 `&` 时它才显露真身（`int[4]` / `int(*)[4]`）。

### 量子类比（帮助你直觉理解）

```
电子能级跃迁：                   指针/数组运算：
E3 ────                           arr[3] ────
     ↕ 只能吸收恰好等于                ↕ arr+3 只能跳
     │ 能级差的能量                    │ 3×sizeof(int) 字节
E2 ────                           arr[2] ────
     ↕ 不能跳半级                    ↕ 不能跳到 arr[1.5]
E1 ────                           arr[1] ────
```

**共同点：存储/状态的粒度是离散的。**
- 物理：没有 E1.5 这个能级
- 指针：没有 `arr[1.5]` 这个地址——元素是离散的，指针步长正好对齐这个离散性

所以指针**必须有类型**——类型决定了"一步跨多大"（int* 跨 4 字节，double* 跨 8 字节），而数组元素正好是等距对齐的，这才能用指针算术遍历。

```
p + N = 地址 + N × sizeof(指向的类型)
```

这不只是公式，它是**数组离散性 + 指针类型系统**共同决定的结果。

## 二、C 风格字符串

### `char[]` vs `const char*`

```cpp
char str1[] = "hello";       // 栈上数组，可读写
const char* str2 = "hello";  // 指针在栈上，字符串在只读区
```

```
内存：
栈：
  str1 = [h][e][l][l][o][\0]    ← 6 字节，可改
  str2 = 0x400800 ──────────┐
                             ↓
数据段（只读）：
  "hello\0" 在 0x400800    ← 只读，不能改！
```

### 关键区别

| | `char str1[] = "hello"` | `const char* str2 = "hello"` |
|:--|:-----------------------|:---------------------------|
| 内存位置 | 栈上 | 指针在栈，字符串在数据段（只读） |
| 能否改内容 | ✅ `str1[0] = 'H';` | ❌ `str2[0] = 'H';` 崩溃 |
| 能否改指向 | ❌ `str1++` 不行 | ✅ `str2 = "world";` |
| `sizeof` | 6（含结尾 `\0`） | 8（指针大小） |
| 字符串存哪 | 栈上拷贝了一份 | 指向数据段的只读常量 |

### 隐藏的 `\0`

```
"hello" 在内存里是：
[h][e][l][l][o][\0]
                ↑ C 风格字符串结尾标志
strlen("hello") = 5
sizeof("hello") = 6（含 \0）
```

## 三、`new`/`delete` vs `malloc`/`free`

### new 干了什么？

```cpp
Test* t = new Test();
// 等价于（伪代码）：
Test* t = (Test*)malloc(sizeof(Test));  // ① 分配内存
t->Test();                                // ② 调用构造函数
```

### delete 干了什么？

```cpp
delete t;
// 等价于（伪代码）：
t->~Test();       // ① 调用析构函数
free(t);          // ② 释放内存
```

### 核心区别

| | `new` / `delete` | `malloc` / `free` |
|:---|:----------------|:-----------------|
| 语言 | C++ | C |
| 调用构造函数 | ✅ 自动调 | ❌ 不调 |
| 调用析构函数 | ✅ 自动调 | ❌ 不调 |
| 返回类型 | 类型安全（`Test*`） | `void*` 需手动转型 |
| 失败行为 | 抛异常 | 返回 `nullptr` |
| 指定大小 | 不需要（编译器知道） | 需要 `sizeof(Test)` |
| 混用 | ❌ `malloc` + `delete` 未定义行为 | ❌ `new` + `free` 未定义行为 |

### AI Infra 的实际做法：内存池

```
不这样：
for (每次推理) {
    float* buf = new float[1024];  // 每次分配
    // ...
    delete[] buf;                  // 释放
}

而是这样：
class MemoryPool {
    char* pool;          // 启动时 malloc 一大块
    size_t offset = 0;

    void* alloc(size_t n) {
        void* p = pool + offset;  // 指针移动
        offset += n;
        return p;
    }
    void reset() { offset = 0; }  // 整个池重用
};
```

推理框架（vLLM、llama.cpp）**启动时一次性分配大块内存**（`malloc`/`cudaMalloc`/`mmap`），后续用指针偏移来"分配"，避免频繁 `new`/`delete` 的性能开销。

## 四、三种指针坑

### ① 野指针（Wild Pointer）

**定义：** 声明了但没初始化的指针。

```cpp
int* p;     // 野指针
*p = 42;    // ❌ 往随机地址写值，可能崩溃或破坏数据
```

**修复：**
```cpp
int* p = nullptr;       // ✅ 初始化
int* p = new int(42);   // ✅ 直接指向有效内存
```

### ② 悬空指针（Dangling Pointer）

**定义：** 指向的内存已释放，但指针还在用。

```cpp
int* p = new int(42);
delete p;      // 堆内存释放了
*p = 100;      // ❌ p 还指向那块地址，但内存可能已被别人占用
```

**为什么危险：**
```
delete 后：
  堆上内存已归还系统，但 p 还存着旧地址
  如果这块内存被别的 new 拿走了 → 你在改别人的数据
  如果还没被分配 → 看起来正常但数据不对
  表现不稳定 → 难以排查的 bug
```

**修复：**
```cpp
delete p;
p = nullptr;  // ✅ 释放后置空，再 *p 就立即崩溃（比默默改坏数据好）
```

### ③ 内存泄漏（Memory Leak）

**定义：** `new` 了但忘记 `delete`。

```cpp
void func() {
    int* p = new int[1000000];  // 分配 4MB
    // 忘记 delete[] p
}  // 函数结束，指针销毁，但堆上 4MB 永远没人能释放了
```

**后果：** 程序内存持续增长，最终 OOM（Out Of Memory）被系统杀掉。

**修复：**
```cpp
void func() {
    int* p = new int[1000000];
    // ... 使用 ...
    delete[] p;           // ✅ 配对释放
}

// 更好的方式：RAII 容器
void func() {
    std::vector<int> p(1000000);  // ✅ 出作用域自动释放
}
```

### 三种坑对比

| 类型 | 本质 | 代码 | 修复 |
|:----|:----|:----|:----|
| 野指针 | 未初始化就开始用 | `int* p; *p = 42;` | 声明时初始化 |
| 悬空指针 | 释放后还在用 | `delete p; *p = 42;` | 释放后置空 |
| 内存泄漏 | 只 new 不 delete | `new int;` 不写 `delete` | 每个 new 配 delete |

### 这些跟 Rust 的关系

Rust 语言诞生的核心动机之一：**在编译时解决 C++ 的内存安全问题。**

```
C++：                    Rust：
int* p = new int(42);     let p = Box::new(42);
delete p;                 drop(p);
*p = 100;  // ❌ 运行时   // *p = 100; ❌ 编译错误！
           // 可能崩溃    // Rust 在编译期就不让用
```

Rust 的所有权系统、借用检查器在**编译阶段**就阻止了野指针、悬空指针和数据竞争——不需要程序员自己小心，也不依赖垃圾回收。

## 五、C++ 最佳实践

1. **优先栈对象**：能不 `new` 就不 `new`，栈自动管理生命周期
2. **RAII 容器**：`vector`、`string`、智能指针自动管理堆内存
3. **必须 new 时**：立即配对写 `delete`，释放后置空
4. **理解内存池**：框架里不会频繁 new，而是一次分配、指针偏移、整体重置
