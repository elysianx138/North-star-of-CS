# DAY2 — 内存布局：栈 / 堆 / 代码段 / 数据段

> **阅读前提：** 需要了解基础的 C++ 语法以及指针基本用法。
>
> **核心主题：** 一个程序跑起来后，内存里到底是怎么划分的？你的变量住哪个区？

---

## 一、程序的内存地图

```
高地址
  ┌─────────────────────┐
  │       栈 (Stack)      │ ← 局部变量、函数调用（向下生长）
  │         ↓↓           │
  │                      │
  │         ↑↑           │
  │       堆 (Heap)       │ ← new/malloc 出来的（向上生长）
  ├─────────────────────┤
  │   数据段 (Data)       │ ← 全局变量、static 变量
  │   ├ 已初始化数据区     │ ← int a = 42;
  │   └ BSS 区            │ ← int a[1000000];（启动时自动清零）
  ├─────────────────────┤
  │   代码段 (Code/Text)  │ ← 你写的那些指令（movl $42, call…）
  └─────────────────────┘
低地址
```

## 二、栈 vs 堆：本质区别

### 实验 1：看看各区域的地址

运行 `./memory`，观察输出的地址大小关系。

```cpp
int global_var = 999; // 数据段

void stackFunc() {
    int localA = 42;      // 栈
    int localB = 100;     // 栈（紧挨 localA）
    std::cout << "栈上 localA 的地址: " << &localA << std::endl;
    std::cout << "栈上 localB 的地址: " << &localB << std::endl;
}

int main() {
    int* p = new int(777); // p 在栈上，但 777 存在堆上
    std::cout << "全局变量 global_var 的地址: " << &global_var << std::endl;
    std::cout << "栈上指针 p 的地址: " << &p << std::endl;
    std::cout << "堆上 777 的地址: " << p << std::endl;
    stackFunc();
    delete p;
    return 0;
}
```

**输出示例（地址大小关系已验证）：**

| 变量 | 地址 | 所在区域 |
|:----|:----|:--------|
| `globalVar` | `0x5a000c17f010` | 数据段（低地址） |
| `*p`（777） | `0x5a003dd7e020` | 堆 |
| `&p` | `0x7ffc6763af00` | 栈（高地址） |
| `localA` | `0x7ffc6763aee0` | 栈 |
| `localB` | `0x7ffc6763aee4` | 栈 |

### 关键问题：`&p` 与 `p` 的区别

```
内存里的样子：

        栈上（地址 0x7ffc6763af00）
        ┌──────────────────────┐
        │   p = 0x5a003dd7e020 │  ← p 这个变量本身占 8 字节
        └──────────────────────┘
                   │
                   │ 指向
                   ▼
        堆上（地址 0x5a003dd7e020）
        ┌──────────────────────┐
        │   777                │  ← 真正的数据
        └──────────────────────┘
```

- `p` → 存的是堆地址 `0x5a003dd7e020`
- `&p` → p 这个变量自己在**栈上的地址**
- `*p` → 去那个地址取值 = `777`

> 指针就是一个变量，在栈上，里面存着堆的地址。

## 三、函数调用时栈发生了什么

运行 `./stack_trace`，观察 `&a`、`&b`、`&c` 的地址大小关系。

```cpp
void func2() { int c = 3; std::cout << &c << std::endl; }
void func1() { int b = 2; std::cout << &b << std::endl; func2(); }
int main()  { int a = 1; std::cout << &a << std::endl; func1(); }
```

### 栈帧布局（叠盘子理论）

```
高地址
  ┌──────────────────────┐
  │   main 的栈帧         │
  │     &a               │  ← 最大地址
  ├──────────────────────┤
  │   func1 的栈帧        │
  │     &b               │  ← 中间
  ├──────────────────────┤
  │   func2 的栈帧        │
  │     &c               │  ← 最小地址
  └──────────────────────┘
低地址
```

**每次调用函数，栈就往下"叠一层"。函数结束后，指针弹回去。**

### 为什么相邻变量差 20 字节？（不只是 4）

一个栈帧里除了局部变量，还有：

```
func1 的栈帧：
  ┌──────────────────────────┐
  │   返回地址 (8字节)        │ ← 告诉 CPU "func1 执行完回哪"
  ├──────────────────────────┤
  │   旧的 rbp (8字节)        │ ← 保存上一个函数的栈底
  ├──────────────────────────┤
  │   局部变量 b (4字节)      │ ← 你的 int b
  ├──────────────────────────┤
  │   对齐填充 (几字节)       │ ← 编译器为了对齐加的填充
  └──────────────────────────┘
  总共 ≈ 20 字节
```

> ⚠️ 函数调用有**隐藏开销**：保存返回地址、保存旧栈底、分配和回收栈帧。

## 四、堆比栈慢了多少？

### 实验 2：一次大分配

运行 `./compare_stack_heap`

| 区域 | 分配 + 遍历 10M 个 int | 耗时 |
|:----|:---------------------|:----|
| 栈 | `int arr[N];` | ~2 ms |
| 堆 | `int* arr = new int[N];` | ~5 ms |

**结论：** 栈 ≈ 2 倍快。

### 实验 3：反复小分配 × 100 万次

运行 `./compare`

| 区域 | 操作 100 万次 | 耗时 |
|:----|:------------|:----|
| 栈 | 反复创建销毁 `Obj o;` | ~1 ms |
| 堆 | 反复 `new Obj / delete` | ~16 ms |

**结论：** 栈 ≈ **16 倍快**。

### 为什么差这么多？

| | 栈 | 堆 |
|:--|:---|:---|
| **分配** | `subq $4, %rsp`——1 条 CPU 指令 | 查空闲链表 → 找块 → 切块 → 更新 → 返回 |
| **释放** | 函数结束，`rsp` 移回去 | 标记空闲 → 合并相邻块 |
| **缓存** | 连续内存，cache 友好 | 不一定紧挨常用数据 |
| **本质** | **一根指针上下移动** | **链表 + 查找 + 碎片管理** |

## 五、在汇编中验证

```bash
g++ -S memory.cpp -o memory.s
```

打开 `memory.s`，找到：

```asm
栈分配：   subq $32, %rsp      ← 1 条指令，栈指针下移
堆分配：   call _Znwm@PLT      ← 调用堆管理器，内部几十上百条指令
```

**一眼就看出为什么栈快——一条指令 vs 一个函数调用。**

## 六、代码段 & 数据段

### 代码段

内存里存你的 **函数指令**（.s 里的 `movl`、`call`、`ret` 翻译成的二进制）。

**关键特性：**
- **只读**——运行时不能修改，改了崩溃
- **大小固定**——编译完就定了
- **共享**——开两个终端跑同一程序，共用一份代码段，省内存

### 数据段

存 **全局变量** 和 `static` 变量。分两部分：

#### ① 已初始化数据区

```cpp
int c = 42;      // 编译时在 exe 里写死了这个 42
```

#### ② BSS 区（未初始化数据区）

```cpp
int a[1000000];  // 没初始化 → BSS
```

**验证 BSS：**

```cpp
// test_bss.cpp
int arr1[1000000];
int main() { return 0; }
```

```cpp
// test_data.cpp
int arr2[1000000] = {0};
int main() { return 0; }
```

编译对比：

```bash
g++ Day2/test_bss.cpp -o Day2/test_bss
g++ Day2/test_data.cpp -o Day2/test_data
ls -lh Day2/test_bss Day2/test_data
```

**结果：**

| 版本 | exe 大小 | 原因 |
|:----|:--------|:-----|
| `arr1[1000000]`（BSS） | **~16KB** | exe 里只记了一行笔记："此需 4MB，启动清零" |
| `arr2[1000000] = {0}`（数据段） | **~4MB** | exe 里实实在在存了 400 万个 0 |

**BSS 的精髓：文件里不存，运行时照占。**
- 运行时两者都占用 **4MB 内存**
- 但 BSS 不占 exe 空间，启动时由操作系统自动填 0
